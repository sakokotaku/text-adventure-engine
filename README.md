# text-adventure-engine


## 这个项目怎么来的

起因很简单：想省钱。用订阅制调用AI玩文字冒险觉得贵，想试试直接调API会不会划算——最后发现并不会。但引擎已经做出来了。

本项目所有代码由AI（Claude + GPT-4）生成，作者全程零代码输入，担任项目经理角色。某种意义上，这个项目本身也是一次关于"一个完全不懂编程的人能否只靠调度AI完成一个完整软件项目"的实验。

## 设计思路

核心问题是：如何让AI在多次对话之间保持一致的世界观、角色性格和关系记忆，同时控制token消耗？

解决方向是把信息分层：不变的东西（规则、角色性格、世界设定）固定在本地文件里，每次只把真正动态的内容推给API。存档不是简单的对话记录，而是一份结构化的"关系快照"，让AI每次读取后能准确还原当前状态。

## 可能的技术亮点

**分层Context架构**
 静态规则文件与存档模板保留在本地作为缓存前缀，每次只向API推送动态上下文，降低token消耗。

**结构化关系存档**
 存档模板包含first_reactions字段，专门记录角色之间已建立的默契与称呼习惯，并明确标注"不得重置为初见反应"——用数据结构强制锁定AI的关系记忆，尝试解决大模型跨session失忆问题。

**骰子系统 + 张力值机制**
 引入D6/D10/D20/D100骰子判定影响事件走向，同时维护持续累积的张力值，张力值到达阈值时触发突发事件，让叙事产生自然起伏而不是线性推进。

**场景锁定规则**
 AI有自动跳场景、自动推进时间的倾向，引擎通过SCENE LOCK / TIME LOCK规则文件将其强制约束，确保叙事连续性。这是踩坑之后得到的解法。

## 注意
规则文件中包含格式约束（如段落空行输出规范），实测DeepSeek对部分格式指令遵守度较差，建议使用Claude或GPT-4系列模型以获得最佳体验。

个人配置规则和引擎时以GPT-5.3instant与Claude Sonnet 4.6的官方网站测试运行，因此在这两个模型可能prompt遵循效果最佳。

## 其他模型的测试
kimi k2.5，输出段落信息非常密集，阅读困难。

qwen3.5-27b，测试第一轮即PASS，游戏预设角色A送角色B回家了，qwen显示送回家，随后角色B又出现在角色A家中。

DeepSeek V4，第一个登场角色永远姓林/方，即便rule要求百家姓；以及第一个登场女主一直是理智、冷静型女主。

## 结论
作为个人技术尝试，它是完整的。作为省钱方案，它失败了。

这只是一个外行人的摸索记录，设计上肯定有大量可以优化的地方，也可能存在运行bug。如果你恰好路过，希望能给你一点参考，见笑了。
